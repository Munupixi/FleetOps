## 3.1 Назначение документа

Настоящий документ определяет основные архитектурные решения, структуру информационной системы FleetOps,
состав основных сущностей предметной области, справочников, перечислений и принципы организации программной системы.

Документ является основой для дальнейшего проектирования базы данных,
REST API и реализации программного обеспечения.

## 3.2 Архитектура системы

FleetOps представляет собой серверное приложение, построенное по архитектуре REST API.

Система обеспечивает централизованное хранение данных, обработку бизнес-логики и
предоставление интерфейса взаимодействия через HTTP-запросы.

При разработке используется многослойная архитектура, обеспечивающая разделение ответственности
между компонентами системы.

## 3.3 Основные подсистемы

В ходе проектирования были выделены следующие подсистемы.

- управление пользователями;
- управление транспортными средствами;
- управление водителями;
- управление маршрутами;
- управление рейсами;
- управление техническим обслуживанием;
- управление ремонтами;
- управление расходом топлива;
- управление документами;
- управление уведомлениями;
- формирование отчетности.

## 3.4 Основные сущности предметной области

В процессе проектирования информационной системы FleetOps были определены основные сущности предметной области.

Разработка системы разделена на три версии. Каждая последующая версия расширяет функциональные возможности предыдущей.

### Версия 1 (MVP)

Первая версия содержит минимальный набор сущностей, необходимых для функционирования системы управления автопарком.

| Сущность | Описание |
|:----------|:---------|
| **User** | Пользователь информационной системы. Используется для аутентификации и выполнения операций в соответствии с назначенной ролью. |
| **Role** | Роль пользователя, определяющая права доступа к функциям системы. |
| **Driver** | Водитель организации. Является пользователем системы и выполняет рейсы на закрепленных транспортных средствах. |
| **Vehicle** | Транспортное средство организации. Хранит основные сведения об автомобиле. |
| **VehicleAssignment** | История закрепления транспортного средства за водителем. Позволяет отслеживать изменения назначений автомобилей. |
| **Route** | Маршрут движения транспортного средства, состоящий из последовательности контрольных точек. |
| **RoutePoint** | Отдельная контрольная точка маршрута с указанием порядка прохождения. |
| **Trip** | Рейс транспортного средства. Содержит информацию о маршруте, автомобиле, водителе, времени выполнения и текущем состоянии рейса. |
| **FuelRecord** | Запись о заправке транспортного средства, содержащая сведения о количестве, стоимости и типе топлива. |
| **Maintenance** | Плановое техническое обслуживание транспортного средства. |
| **Repair** | Ремонт транспортного средства после возникновения неисправности. |

### Версия 2

Во второй версии планируется расширение функциональности системы за счет хранения документов, файлов, уведомлений и учета дополнительных расходов.

| Сущность | Описание |
|:----------|:---------|
| **Document** | Документ, относящийся к транспортному средству или водителю (ПТС, СТС, ОСАГО, диагностическая карта и др.). |
| **Attachment** | Файл, прикрепленный к объектам системы (документам, ремонтам, ДТП, рейсам и другим сущностям). |
| **Expense** | Дополнительные расходы, связанные с эксплуатацией транспортного средства, не относящиеся к ремонту и заправкам. |
| **Supplier** | Организация, поставляющая топливо, запасные части или оказывающая услуги по обслуживанию транспортных средств. |
| **Notification** | Уведомление пользователя о событиях, требующих внимания. |

### Версия 3

Третья версия предусматривает реализацию расширенных возможностей, характерных для корпоративных систем управления автопарком.

| Сущность | Описание |
|:----------|:---------|
| **Inspection** | Результат осмотра транспортного средства перед рейсом, после рейса либо в процессе технической диагностики. |
| **Checklist** | Контрольный список, используемый при проведении осмотра транспортного средства. |
| **Accident** | Дорожно-транспортное происшествие с участием транспортного средства. |
| **Fine** | Штраф, полученный транспортным средством или водителем. |
| **ServiceStation** | Станция технического обслуживания, выполняющая ремонт и обслуживание транспортных средств. |
| **AuditLog** | Журнал действий пользователей, предназначенный для хранения истории изменений, выполненных в системе. |

## 3.5 Основные связи между сущностями

Между основными сущностями предметной области существуют следующие связи.

| Первая сущность | Вторая сущность | Тип связи |
|:----------------------|:------------------|:---------:|
| **User** | **Role** | многие к одному |
| **User** | **Driver** | один к одному |
| **Driver** | **VehicleAssignment** | один ко многим |
| **Vehicle** | **VehicleAssignment** | один ко многим |
| **Route** | **RoutePoint** | один ко многим |
| **Route** | **Trip** | один ко многим |
| **Driver** | **Trip** | один ко многим |
| **Vehicle** | **Trip** | один ко многим |
| **Vehicle** | **FuelRecord** | один ко многим |
| **Vehicle** | **Maintenance** | один ко многим |
| **ServiceStation** | **Maintenance** | один ко многим |
| **Vehicle** | **Repair** | один ко многим |
| **ServiceStation** | **Repair** | один ко многим |
| **Driver** | **Document** | один ко многим |
| **Vehicle** | **Document** | один ко многим |
| **Document** | **Attachment** | один ко многим |
| **Vehicle** | **Expense** | один ко многим |
| **Supplier** | **Expense** | один ко многим |
| **User** | **Notification** | один ко многим |
| **Vehicle** | **Inspection** | один ко многим |
| **Inspection** | **Checklist** | один ко многим |
| **Vehicle** | **Accident** | один ко многим |
| **Driver** | **Accident** | один ко многим |
| **Vehicle** | **Fine** | один ко многим |
| **Driver** | **Fine** | один ко многим |
| **User** | **AuditLog** | один ко многим |

> **Примечание 1.**
> Связь между сущностями **Driver** и **Vehicle** логически является связью «многие ко многим».
Для хранения истории закрепления транспортных средств за водителями она реализована через промежуточную сущность
**VehicleAssignment**.

> **Примечание 2.**
> Сущность **Document** может относиться как к транспортному средству, так и к водителю.
Конкретный владелец документа определяется соответствующей связью.

> **Примечание 3.**
> Сущность **Attachment** предназначена для хранения файлов, прикрепляемых к объектам системы.
На текущем этапе проектирования файлы прикрепляются к документам.
В дальнейшем механизм может быть расширен для поддержки других сущностей без изменения общей архитектуры системы.

> **Примечание 4.**
> Сущность **Driver** представляет собой расширение учетной записи пользователя и содержит сведения,
относящиеся непосредственно к водителю. Не каждый пользователь системы является водителем.

## 3.6 Справочники

Для хранения изменяемых справочных данных в системе используются отдельные таблицы-справочники.

Использование справочников позволяет избежать дублирования данных, обеспечить единообразие значений и
упростить расширение системы без изменения программного кода.

### Версия 1

| Справочник | Назначение |
|:-----------|:-----------|
| **MaintenanceType** | Содержит перечень видов технического обслуживания транспортных средств. |
| **RepairType** | Содержит перечень видов ремонтов транспортных средств. |
| **ExpenseType** | Содержит перечень видов дополнительных расходов. |

### Версия 2

Во второй версии отдельные таблицы-справочники не планируются.

### Версия 3

В третьей версии отдельные таблицы-справочники не планируются.

Состав справочников может быть расширен в последующих версиях системы без изменения архитектуры приложения.

## 3.8 Основные бизнес-правила

При проектировании системы приняты следующие бизнес-правила.

- Один пользователь может иметь только одну роль.
- Не каждый пользователь системы является водителем.
- Один водитель может выполнять множество рейсов.
- Один рейс выполняется одним водителем на одном транспортном средстве.
- Одно транспортное средство может использоваться разными водителями в разные периоды времени.
- История закрепления транспортных средств хранится в сущности VehicleAssignment.
- Каждый маршрут должен содержать не менее двух контрольных точек: начальную и конечную.
- Один документ относится либо к водителю, либо к транспортному средству.
- Каждая запись о заправке относится к одному транспортному средству.
- Каждое техническое обслуживание относится к одному транспортному средству.
- Каждый ремонт относится к одному транспортному средству.
- Все действия пользователей фиксируются в журнале аудита.

## 3.9 Ограничения целостности данных

При проектировании базы данных предусматриваются следующие ограничения.

- Каждая запись имеет уникальный идентификатор.
- Внешние ключи должны ссылаться только на существующие записи.
- Удаление записей не должно приводить к нарушению ссылочной целостности.
- Для обязательных полей запрещается хранение пустых значений.
- Значения перечислений должны принадлежать допустимому набору.
- История закрепления автомобилей не должна содержать пересекающихся периодов для одного транспортного средства.

## 3.10 Принципы проектирования

При разработке FleetOps используются следующие принципы проектирования.

- модульность;
- разделение ответственности;
- слабая связанность компонентов;
- высокая связность модулей;
- расширяемость архитектуры;
- повторное использование компонентов;
- соблюдение принципов SOLID;
- единый источник хранения данных.
