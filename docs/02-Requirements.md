## 2.1 Назначение документа

Настоящий документ определяет требования к информационной системе FleetOps.

Требования являются основой для проектирования архитектуры системы, структуры базы данных, REST API и
реализации программного обеспечения.

Все требования, приведенные в настоящем документе, должны учитываться при дальнейшей разработке проекта.

## 2.2 Назначение системы

Система FleetOps предназначена для автоматизации процессов управления автопарком предприятия.

Основной задачей системы является централизованное хранение информации, автоматизация бизнес-процессов и
предоставление инструментов контроля эксплуатации транспортных средств.

## 2.3 Цели системы

Разрабатываемая система должна обеспечивать:

- централизованное хранение информации;
- сокращение количества ручных операций;
- уменьшение вероятности возникновения ошибок;
- повышение эффективности управления автопарком;
- повышение качества хранения данных;
- автоматизацию основных бизнес-процессов;
- получение актуальной аналитической информации;
- возможность дальнейшего расширения функциональности.

## 2.4 Категории пользователей

Система должна поддерживать работу следующих категорий пользователей.

### Руководитель

Руководитель использует систему для контроля деятельности организации, анализа эффективности эксплуатации транспорта и
получения аналитической информации.

Руководитель должен иметь возможность:

- просматривать статистику;
- получать отчеты;
- контролировать деятельность подразделений;
- анализировать эффективность работы автопарка.

### Диспетчер

Диспетчер осуществляет ежедневное управление эксплуатацией транспорта.

Диспетчер должен иметь возможность:

- регистрировать транспортные средства;
- управлять маршрутами;
- назначать транспорт;
- назначать водителей;
- оформлять рейсы;
- контролировать выполнение перевозок.

### Водитель

Водитель использует систему при эксплуатации закрепленного транспортного средства.

Водитель должен иметь возможность:

- просматривать закрепленный автомобиль;
- просматривать информацию о маршруте;
- фиксировать пробег;
- сообщать о неисправностях.

### Механик

Механик отвечает за техническое состояние транспорта.

Механик должен иметь возможность:

- регистрировать техническое обслуживание;
- оформлять ремонт;
- просматривать историю обслуживания;
- контролировать техническое состояние автомобилей.

### Бухгалтер

Бухгалтер использует систему для учета расходов.

Бухгалтер должен иметь возможность:

- учитывать расходы;
- учитывать стоимость ремонтов;
- учитывать расходы топлива;
- получать финансовые отчеты.

### Администратор

Администратор обеспечивает сопровождение информационной системы.

Администратор должен иметь возможность:

- управлять пользователями;
- управлять ролями;
- изменять справочную информацию;
- контролировать корректность данных.

## 2.5 Бизнес-требования

Разрабатываемая система должна обеспечивать возможность:

- ведения единой базы данных автопарка;
- хранения истории эксплуатации транспортных средств;
- хранения истории технического обслуживания;
- хранения истории ремонтов;
- хранения информации о расходе топлива;
- контроля технического состояния автомобилей;
- получения актуальной информации о состоянии автопарка;
- формирования аналитических отчетов;
- поиска необходимой информации;
- снижения количества ручных операций.

## 2.6 Функциональные требования

Система должна обеспечивать следующие функциональные возможности.

### Управление пользователями

Система должна позволять:

- создавать пользователей;
- изменять информацию о пользователях;
- удалять пользователей;
- просматривать список пользователей;
- выполнять поиск пользователей;
- назначать роли пользователям.

### Управление транспортными средствами

Система должна позволять:

- регистрировать транспортные средства;
- изменять сведения о транспортных средствах;
- удалять транспортные средства;
- просматривать список транспортных средств;
- искать транспортные средства;
- просматривать историю эксплуатации автомобиля.

### Управление водителями

Система должна обеспечивать:

- регистрацию водителей;
- изменение информации;
- просмотр информации;
- закрепление транспорта;
- изменение закрепления транспорта.

### Управление техническим обслуживанием

Система должна обеспечивать:

- регистрацию технического обслуживания;
- изменение информации;
- хранение истории обслуживания;
- планирование следующего обслуживания.

### Управление ремонтами

Система должна обеспечивать:

- регистрацию ремонта;
- хранение истории ремонтов;
- регистрацию стоимости ремонта;
- просмотр выполненных ремонтов.

### Управление топливом

Система должна обеспечивать:

- регистрацию заправок;
- учет расхода топлива;
- просмотр истории заправок;
- получение информации о среднем расходе топлива.

### Управление маршрутами

Система должна обеспечивать:

- создание маршрутов;
- изменение маршрутов;
- удаление маршрутов;
- просмотр маршрутов.

### Управление рейсами

Система должна обеспечивать:

- регистрацию рейсов;
- назначение водителя;
- назначение транспортного средства;
- завершение рейса;
- хранение истории рейсов.

### Формирование отчетов

Система должна обеспечивать возможность формирования отчетов по:

- транспортным средствам;
- пробегу;
- техническому обслуживанию;
- ремонтам;
- расходу топлива;
- эксплуатации транспорта.

## 2.7 Нефункциональные требования

Разрабатываемая система должна удовлетворять следующим требованиям.

### Надежность

Система должна обеспечивать сохранность информации при штатной эксплуатации.

### Производительность

Основные операции должны выполняться без заметных задержек для пользователя.

### Масштабируемость

Архитектура должна обеспечивать возможность расширения функциональности без полного изменения структуры системы.

### Поддерживаемость

Код проекта должен иметь модульную структуру и обеспечивать возможность дальнейшего сопровождения.

### Расширяемость

Архитектура должна предусматривать возможность добавления новых модулей.

## 2.8 Требования к безопасности

Система должна обеспечивать:

- аутентификацию пользователей;
- авторизацию на основе ролей;
- безопасное хранение паролей;
- защиту конфиденциальных данных;
- журналирование действий пользователей.

## 2.9 Требования к данным

Система должна обеспечивать:

- целостность данных;
- отсутствие дублирования информации;
- возможность хранения истории изменений;
- централизованное хранение информации.

## 2.10 Критерии успешности

Проект считается успешно реализованным при выполнении следующих условий:

- реализованы все основные функциональные требования;
- обеспечена ролевая модель доступа;
- реализовано централизованное хранение данных;
- система готова к дальнейшему расширению;
- архитектура соответствует требованиям проекта.
